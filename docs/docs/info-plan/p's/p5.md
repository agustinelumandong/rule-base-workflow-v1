# BookForge / CanonForge Reference Pack

## 1. Core Architecture: Stateful AI Workflows

Use these for the **pipeline/state machine** layer.

| Reference                       | Use It For                                                                                                                                                                                                   |
| ------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| **LangGraph Docs**              | Stateful agent workflows, durable execution, human-in-the-loop review, long-running book jobs. LangGraph explicitly supports durable execution, human oversight, and long-term memory. ([LangChain Docs][1]) |
| **LangGraph GitHub**            | Source-level examples for long-running, stateful agents. ([GitHub][2])                                                                                                                                       |
| **LangGraph Durable Execution** | Important for BookForge because book generation can resume without redoing completed steps. ([LangChain Docs][3])                                                                                            |
| **Temporal Python SDK**         | Alternative if you want industrial workflow orchestration instead of LangGraph-native orchestration. Temporal is for long-running, durable workflows and worker-based execution. ([Temporal Docs][4])        |

**My recommendation:** use **LangGraph first**. Consider Temporal later only if BookForge becomes multi-user, queued, distributed, and production-hosted.

## 2. RAG / Retrieval / Context Engineering

Use these for **MemoryVault**, context packets, semantic search, and reference retrieval.

| Reference                     | Use It For                                                                                                                                          |
| ----------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| **LlamaIndex GitHub**         | RAG ingestion, indices, retrievers, query engines, rerankers, document workflows. ([GitHub][5])                                                     |
| **LlamaIndex / IBM Overview** | Good conceptual overview of LlamaIndex as a data orchestration framework for LLM apps and RAG. ([IBM][6])                                           |
| **pgvector GitHub**           | Store embeddings directly inside PostgreSQL. Good first vector layer for BookForge because canon facts also need structured metadata. ([GitHub][7]) |
| **pgvector Python**           | Python integration with SQLAlchemy, SQLModel, Psycopg, asyncpg, Django, etc. ([GitHub][8])                                                          |
| **Qdrant Docs**               | Dedicated vector search engine with filtering, hybrid queries, payload metadata, and scalable retrieval. ([Qdrant][9])                              |
| **Qdrant Filtering**          | Important for filtering by `story_id`, `book_id`, `chapter`, `era`, `character`, `artifact_type`. ([Qdrant][10])                                    |
| **Chroma Docs**               | Easy local vector DB for experimentation; supports embeddings, metadata, dense/sparse retrieval. ([docs.trychroma.com][11])                         |

**My recommendation:** start with **PostgreSQL + pgvector**. Add **Qdrant** later if retrieval grows beyond what Postgres comfortably handles.

## 3. Graph Memory / Series Continuity

Use these for **character relationships, unresolved threads, object state, weapon state, and book-to-book continuity**.

| Reference                          | Use It For                                                                                                                  |
| ---------------------------------- | --------------------------------------------------------------------------------------------------------------------------- |
| **Neo4j GraphRAG Python Docs**     | GraphRAG patterns using Neo4j; useful once CanonCore needs relationship-heavy retrieval. ([Graph Database & Analytics][12]) |
| **Neo4j GraphRAG User Guide**      | Practical RAG guide for graph-backed retrieval. ([Graph Database & Analytics][13])                                          |
| **Neo4j GraphRAG Developer Guide** | Conceptual guide for graph retrieval in GenAI apps. ([Graph Database & Analytics][14])                                      |
| **Kuzu Docs**                      | Embedded graph database. Good if you want local-first graph memory without running a Neo4j server. ([Kuzu][15])             |
| **Kuzu GitHub**                    | Embedded property graph DB with full-text search and vector indices. ([GitHub][16])                                         |

**My recommendation:** do **not** start with Neo4j. Start with relational tables:

```text
characters
relationships
timeline_events
objects
weapon_state
unresolved_threads
```

Then add **Kuzu** or **Neo4j** only when graph queries become painful.

```json

  "weapon": "Colt Paterson",

  "capacity": 5,
  "valid_after": 1836,
  "reload_notes": "slow percussion-era loading"
}
```

Then CanonCore can reject impossible weapon usage.
