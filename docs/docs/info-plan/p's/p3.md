# BOOK FORGE — MASTER BUILD BRIEF

## 1. Product Mission

Build **Book Forge**, a production-grade AI book generation and editing system.

Book Forge should take a story concept, outline, chapter summaries, style requirements, canon rules, and human instructions, then generate or rewrite full books through a controlled pipeline.

The goal is **not** “let AI write randomly.”

The goal is:

> Build a reliable AI-assisted book production factory that can generate, expand, humanize, validate, revise, export, and track books at scale while preserving story canon, character consistency, tone, POV, plot continuity, and human review control.

Book Forge must support both:

1. **Book generation from scratch**

   * Idea → premise → characters → outline → chapter plans → full chapters → revisions → export.

2. **Book transformation / humanization**

   * Existing AI-assisted manuscript → natural Western prose → validation → human review → final book.

---

# 2. Core Principle

Book Forge is **not one prompt**.

Book Forge is a **pipeline system**.

Every book must pass through deterministic stages, where each stage produces artifacts, saves files, validates output, and allows human approval.

The system must avoid dumping entire books into one context window.

Instead, it must use:

* chapter-level processing
* scene-level chunking
* canon memory
* retrieval
* validators
* diffs
* approvals
* versioning
* export tools

# 8. Data Schemas

```python
class ChapterPlan(BaseModel):
```

```md
You are responsible for preserving story canon.

You must:
- detect contradictions
- detect invented facts
- detect changed character traits
- detect timeline errors
- detect location inconsistencies
- detect POV violations
- detect forbidden changes

You must not rewrite prose unless asked.

Return JSON only.

  "passed": true/false,

      "type": "canon|timeline|character|location|pov|style",

      "recommendation": "..."
    }
  ]
}
```

---

## B. Chapter Writer Prompt

```md
# Chapter Writer

You write one chapter at a time using only the provided context.

Inputs:
- story bible
- character sheet
- chapter plan
- scene plan
- prior chapter summary
- style guide
- forbidden changes

Rules:
- Follow the chapter plan.
- Preserve canon.

- Maintain POV.
- Maintain tense.
- Use natural prose.
- Avoid generic AI phrasing.
- Avoid purple prose unless style guide requests it.

Output:
Markdown chapter text only.
```

```md
# Prose Humanizer

You improve naturalness, rhythm, clarity, emotional realism, and Western conversational flow.

Inputs:
- draft text
- style guide
- canon notes
- character voice rules

Rules:
- Preserve all plot facts.
- Preserve POV.
- Preserve tense.
- Preserve character intent.

- Do not remove required events.
- Do not make prose overly dramatic.
- Prefer clean, concrete language.

- Keep meaning unchanged.

Rewritten text only.
```

# 11. RAG / Memory System

Book Forge should use RAG, but carefully.

RAG is not the writer.

RAG is the memory retrieval layer.

Use RAG for:

* retrieving relevant canon
* retrieving character facts
* retrieving prior chapter summaries
* retrieving timeline events
* retrieving style rules
* retrieving glossary terms
* retrieving location descriptions

Do **not** use RAG to blindly stuff random chunks into prompts.

Each stage should request only relevant context.

Example:

```txt
For Chapter 12, retrieve:
- current chapter plan
- protagonist profile
- antagonist profile
- location facts
- previous 2 chapter summaries
- unresolved plot threads
- required future constraints
```

# 12. Context Strategy

Never send the whole book unless absolutely necessary.

Use this hierarchy:

```txt
Book Bible
  → Act Summary
    → Chapter Summary
      → Scene Plan
        → Scene Draft
```

For generation:

* Generate one scene or chapter at a time.
* Validate after each scene/chapter.
* Summarize completed chapters.
* Store summaries in memory.
* Retrieve only relevant facts for the next chapter.

For long manuscripts:

* chunk by scene
* preserve boundaries
* process chunks
* reassemble
* validate chapter-level continuity
* validate book-level continuity later

# 19. Local vs Cloud Model Strategy

Use hybrid.

## Local Models

Good for:

* cheap testing
* summaries
* first-pass validation
* formatting
* repetition checks
* simple rewrite experiments
* private/offline workflow

## Cloud Models

Better for:

* high-quality prose
* complex canon reasoning
* final humanization
* final revision
* difficult continuity validation

Recommendation:

> Build Book Forge model-agnostic. Start with local + cheap APIs for development. Use premium models only for high-value stages.
