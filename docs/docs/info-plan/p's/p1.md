## 3. Durable Workflow Engine

For this, I would seriously consider **LangGraph**.

LangGraph is designed for long-running, stateful agent workflows, and its durable execution layer can persist workflow state so completed steps do not need to be recomputed after delays or failures. ([LangChain Docs][1])

That matters for you because book generation is not one call.

It is a long-running production process:

```text
Book Job
  Chapter Jobs
    Scene Jobs
      Draft
      Validate
      Evaluate
      Repair
      Approve
```

A proper workflow graph lets you say:

```text
If context fails → context repair
If style fails → style repair
If reader score fails → narrative revision
If length short → source-supported expansion
If all pass → compile
```

That is much cleaner than asking an AI agent to decide everything from an instruction file.

## Vector Search

Start with:

```text

```

Do not overcomplicate too early.

For your use case, pgvector is a strong first choice because your canon memory is not just semantic text. It is also structured data: book ID, series ID, chapter number, character, timeline position, source artifact type, approval status.

A recent RAG data-layer paper specifically argues that unified PostgreSQL + vector search reduces synchronization issues and tenant/data leakage risks compared with split systems. ([arXiv][2])

My recommendation:

```text
Phase 1: PostgreSQL + pgvector
Phase 2: Add Qdrant only if retrieval scale/performance demands it
```

Do not start with Pinecone unless you want managed cloud and are okay with cost/vendor dependency.

# RAG: Yes, But Not Dumb RAG

You absolutely should use retrieval.

But not generic “chunk manuscript and search it” RAG.

For fiction, you need **structured canon retrieval**.

There are three memory types:

## 1. Canon Memory

Hard truth.

Examples:

```text
Character: Jed
Missing fingers: yes
Cause: old injury
Relationship: distrustful of strangers
```

This should live in PostgreSQL as structured records.

## 2. Narrative Memory

Story movement.

Examples:

```text
Chapter 12:
Jed discovers sheriff corruption.
Clementine becomes more active.
Town pressure increases.
```

This can live as markdown artifacts plus embeddings.

## 3. Style Memory

Reusable prose patterns.

Examples:

```text
Classic Western opening pattern
Dark Western violence rhythm
Dialogue density examples
Scene transition examples
```

This can live in vector search.

# Knowledge Graph: Strongly Recommended

For series-level continuity, vector search alone is not enough.

You need a **canon graph**.

Example:

```text
Jed --knows--> Clementine
Clementine --betrayed_by--> Sheriff
Sheriff --controls--> town deputies
Gold cache --located_near--> Bitterroot trail
Book 1 ending --sets_up--> Book 2 antagonist
```

This lets the system answer:

* Who knows this secret?
* Who is dead?
* Who has met whom?
* What promises are unresolved?
* What must carry into Book 2?
* What should not be contradicted?

This is critical for book-by-book continuation.

I would store graph-like relationships first in PostgreSQL tables, not necessarily Neo4j on day one.

Example:

```text
canon_entities
canon_relationships
timeline_events
unresolved_threads
series_promises
```

Later, if graph queries get complex, add Neo4j or Kuzu.

# MCP: Use It, But Carefully

MCP is useful, but it should not be the heart of the system.

The Model Context Protocol exposes tools, resources, and prompt templates to models. Its official spec defines server-side features like resources, prompts, and tools, and MCP tools allow models to interact with external systems such as databases, APIs, and computations. ([Model Context Protocol][4])

For CanonForge, MCP is useful for:

* accessing manuscript files
* querying canon DB
* running validators
* searching references
* calling export tools
* retrieving style guides

But I would not let MCP become “AI can call everything.”

Limit tools by mode.

Example:

```text
Drafting Agent tools:
- get_scene_card
- get_relevant_canon
- save_draft

Validation Agent tools:
- get_draft
- get_canon
- report_failure

Compiler Agent tools:
- read_approved_chapters
- compile_manuscript
```

Never expose unnecessary tools.

Tool overload makes agents dumber, not smarter.

```text
Chapter Summary
→ Scene Breakdown
→ Draft
```

New flow:

```text
Chapter Summary
→ Scene Breakdown
→ Scene Cards

→ Validate Scene
→ Approve Scene
```

A scene card should include:

```text
Scene ID:
chapter-07-scene-02

POV:
Jed

Location:
Livery stable

Time:
Night after town confrontation

Purpose:
Force Jed to realize the sheriff's men are watching him.

Required Facts:
- Clementine warns him indirectly.
- Deputy appears outside.
- Jed does not confront him yet.

Conflict:
Jed wants to leave quietly, but the town is closing around him.

Emotional Turn:
Caution becomes resolve.

Exit Hook:
Jed finds evidence someone touched his saddle.
```

This gives the model less room to hallucinate.

It also gives validators something precise to check.
