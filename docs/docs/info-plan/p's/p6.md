# Book Forge Reference Pack

## 1. Core Backend Stack

Use these as the foundation.

| Area            | Technology                            | Why it matters                                                                                                                                                                           |
| --------------- | ------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| API backend     | **FastAPI**                           | Clean Python API framework, good for local and hosted app modes. FastAPI supports SQL/database integration and is commonly used with Pydantic-style schemas. ([fastapi.tiangolo.com][1]) |
| Data validation | **Pydantic v2**                       | Required for strict schemas: projects, chapters, canon, validation reports, model calls, approvals. ([Pydantic][2])                                                                      |
| Database        | **SQLite first, PostgreSQL later**    | SQLite for MVP; Postgres when multi-user/team scale is needed.                                                                                                                           |
| ORM             | **SQLAlchemy / SQLModel**             | Useful for structured database models. FastAPI’s SQL docs use SQLModel, built on SQLAlchemy and Pydantic. ([fastapi.tiangolo.com][1])                                                    |
| CLI             | **Typer**                             | Build commands like `bookforge init`, `bookforge write-chapter`, `bookforge export`.                                                                                                     |
| Config          | **python-dotenv / pydantic-settings** | Manage `.env`, API keys, model settings, costs, stage configs.                                                                                                                           |

## 2. AI Provider References

Book Forge should be **model-agnostic**. Your agent should design a provider abstraction from the start.

| Provider / Tool          | Use                                                                                                                                                                                                                  |
| ------------------------ | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **OpenAI API**           | High-quality writing, structured outputs, tool calls, final revision.                                                                                                                                                |
| **Anthropic Claude API** | Strong long-form prose, editorial reasoning, continuity review.                                                                                                                                                      |
| **Google Gemini API**    | Long context, multimodal potential, OpenAI-compatible access is supported by Gemini docs. ([Google AI for Developers][3])                                                                                            |
| **OpenRouter**           | One API for many models, useful for testing multiple providers and fallback routing. OpenRouter says it provides access to major models through a unified interface and works with the OpenAI SDK. ([OpenRouter][4]) |
| **Ollama**               | Local model serving and local development. Ollama supports tool calling and OpenAI-compatible API behavior for some endpoints. ([Ollama Docs][5])                                                                    |
| **llama.cpp**            | Local GGUF inference, useful for cheap testing and private workflows.                                                                                                                                                |
| **vLLM**                 | Production-grade open-model serving if later deployed on GPU server.                                                                                                                                                 |
| **LiteLLM**              | Good abstraction layer if you want one interface across OpenAI, Anthropic, Gemini, OpenRouter, local endpoints, etc.                                                                                                 |

My recommendation: **do not lock Book Forge to LangChain or one provider**. Build your own `BaseLLMProvider` interface, then optionally plug in LiteLLM/OpenRouter/local providers.

## 4. RAG / Memory Stack

Book Forge should use RAG as **canon memory**, not as a magic writing engine.

| Tool           | Best use                                                                                                                                           |
| -------------- | -------------------------------------------------------------------------------------------------------------------------------------------------- |
| **ChromaDB**   | Best for local prototype and MVP RAG. Chroma is lightweight and easy to run locally, making it strong for prototyping. ([AltexSoft][7])            |
| **Qdrant**     | Better production vector database option; strong filtering and deployment flexibility.                                                             |
| **pgvector**   | Best if you want vectors inside PostgreSQL later. Good for simpler ops.                                                                            |
| **FAISS**      | Local-only vector search, good for experiments but less ideal as product storage.                                                                  |
| **LlamaIndex** | Strong for document ingestion, indexing, retrieval workflows.                                                                                      |
| **LangChain**  | Broad ecosystem for LLM apps and agents. LangChain describes itself as a framework for building agents and LLM-powered applications. ([GitHub][8]) |
| **LangGraph**  | More suitable than plain LangChain when you need stateful, multi-step workflow graphs.                                                             |
| **Haystack**   | Stronger production-style RAG pipeline framework.                                                                                                  |
| **FlashRAG**   | Useful research reference; implements advanced RAG methods and benchmark tooling. ([arXiv][9])                                                     |

My recommendation:

```txt
MVP: custom simple RAG layer + ChromaDB.
Production: Qdrant or pgvector.
Do not start with complex agentic RAG.
```

## 6. Diff / Change Tracking

Book Forge needs Git-like visibility.

| Tool                 | Use                                                                                                                                                 |

| **diff-match-patch** | Better text diff/patch algorithms for plain text synchronization. ([PyPI][11])                                                                      |
| **GitPython**        | Optional if you want project folders versioned through Git.                                                                                         |
| **DeepDiff**         | Useful for JSON/YAML artifact differences.                                                                                                          |

Agent instruction:

```txt
Every rewrite must save before.md, after.md, diff.html, and change_summary.json.
```

## 12. Architecture References

Your AI agent should study these patterns:

| Pattern                  | Why                                      |
| ------------------------ | ---------------------------------------- |
| Pipeline architecture    | Book generation must be stage-based.     |
| Artifact-based workflow  | Every stage saves output.                |
| Event sourcing-lite      | Logs every important transition.         |
| State machine            | Prevents invalid stage order.            |
| Repository pattern       | Clean project/file/database access.      |
| Provider adapter pattern | Swap AI models without rewriting system. |
| Human-in-the-loop        | Required for approval gates.             |
| Retry with backoff       | Needed for model/API failure.            |
| Idempotent stages        | Safe re-run without corrupting project.  |

Agent instruction:

```txt
Every stage must be restartable. If a stage fails, the previous artifacts must remain untouched.
```
